# Tool-Kit 02 · Prompt 模板矩阵 · PDCA × 3 族

> 12 个日常 QA prompt，按 PDCA（Plan / Do / Check / Act）× 3 族（F1 覆盖 / F2 回归 / F3 合规）矩阵组织。
> 与 `scenarios/` 不同：scenarios 放长篇深入 prompt；本文件是复制粘贴的速拨表。

## 矩阵总览

|  | Plan | Do | Check | Act |
|--|------|-----|-------|-----|
| **F1 覆盖与单元测试** | §1.1 测试计划骨架 | §1.2 单元测试生成器 | §1.3 覆盖率缺口检测 | §1.4 Fixture 清洗 |
| **F2 回归与变更影响** | §2.1 PR 变更影响地图 | §2.2 回归套件选择 | §2.3 缺陷分流草稿 | §2.4 根因假设链 |
| **F3 合规与审计链路测试** | §3.1 税规测试用例规划 | §3.2 审计链路断言生成器 | §3.3 合规覆盖率审计 | §3.4 QA 周报撰写 |

> 每个格子是一段短 prompt（< 200 词）。`<尖括号>` 内是变量 — 发出前先替换。直接粘贴到 Claude / ChatGPT / Codex。

## F1 · 覆盖与单元测试

### 1.1 P · 测试计划骨架

```
你是一位 QA 工程师，正在为单个特性起草测试计划。

特性说明：
<粘贴特性 PRD 或设计文档，或 Jira/Linear 工单链接文本>

被测代码（变更文件）：
<粘贴文件列表，或 git diff --name-only 输出>

产出一份测试计划，包含以下章节：
- 目标（1 句话，本特性带来什么风险，QA 必须关掉）
- 适用测试类型（unit / integration / E2E — 各选一项并给 1 行理由）
- 风险区（≥ 5 — 明确业务风险，不是技术噪音）
- 测试数据需求（需要的合成 fixture；任何生产形态数据风险都要标注）
- 退出标准（具体：行覆盖 ≥ N%、所有关键路径绿、零 P0 缺陷）
- 范围外（明确不测的项，给理由）

约束：现在还不要写测试用例 — 先做计划。用例在 §1.2。
```

### 1.2 D · 单元测试生成器

```
你是一位 TDD 纪律的测试编写者。

被测函数 / 方法：
<粘贴函数源码>

类型签名 / 接口：
<粘贴相关类型、schema 或 docstring>

产出测试套件（按文件扩展名匹配测试框架 — .py 用 pytest，.ts/.js 用 jest/vitest，.go 用 go testing）：

- Happy path：1 个用例
- 边界用例：≥ 3（off-by-one、空输入、最大尺寸输入、类型边界）
- 错误 / 异常用例：≥ 2（无效输入、依赖失败）
- 幂等性 / 确定性：函数有副作用时 1 个用例
- 测试名要像句子："test_<对象>_<条件>_<预期>"

约束：
- 仅用合成 fixture；不允许真实纳税人识别号 / 银行账号 / 生产 ID
- 每个测试断言一种行为 — 测试名里出现 AND 就拆分
- 用例需要 fixture 时，inline 声明或引用 `tests/fixtures/<name>.json`
```

### 1.3 C · 覆盖率缺口检测

```
你是一位测试覆盖率分析师。

覆盖率报告（行 / 分支）：
<粘贴 coverage / nyc / c8 / jacoco 的输出>

本 PR 变更的源文件：
<粘贴 git diff --name-only>

产出缺口分析：
1. 变更行覆盖率 < 80% 的文件（列当前 %）
2. 变更代码里未覆盖的分支（标行号 + 分支方向）
3. 完全无测试引用的函数（标 file:function）
4. 推荐补全顺序（业务风险最高的先补，每条注明风险）

约束：现在不写测试 — 只标缺口。测试在缺口列表 review 后由 §1.2 出。
```

### 1.4 A · Fixture 清洗

```
你是一位测试数据隐私审计员。

Fixture 文件（粘贴内容或路径）：
<粘贴 fixture JSON/CSV/SQL>

扫描 PII / 生产形态数据：
- 纳税人识别号（18 位模式 91[0-9]{16}|[0-9]{18}）
- 银行账号（16-19 位数字）
- 真实人名比对已知员工列表（如有提供）
- 真实公司名出现在生产租户表里
- 生产发票 ID / 合同 ID

每条命中输出：
| 文件 | 行 | 模式 | 推荐合成替换值 |

约束：
- 不修改文件；只输出命中清单
- 推荐替换值要明显是假的（例：91110000XXXXXXXXXX 带 X 占位，或 "TestCo-001"）
- 零命中则输出 "未检测到 PII" — 不要造假阳性
```

## F2 · 回归与变更影响

### 2.1 P · PR 变更影响地图

```
你是一位代码变更影响分析师。

PR diff：
<粘贴 git diff 或 PR URL>

模块依赖图（如有）：
<粘贴 import graph，或引用变更模块的文件列表>

产出影响地图：
- 直接变更（PR 修改的文件 — 列行数）
- 1-hop 调用方（import 变更代码的模块 — 列文件路径）
- 2-hop 调用方（import 1-hop 调用方的模块 — 若位于关键路径则列出）
- 间接风险：触及的共享类型、共享 fixture、共享工具
- 必须跑的测试套件：按套件名 / 路径枚举
- 可选跑的测试套件：枚举并给理由

约束：除非 PR 改的是被 ≥ 5 个模块用的共享工具，否则不推荐 "全跑一遍"。要具体。
```

### 2.2 D · 回归套件选择

```
你是一位回归测试选择器。

变更影响地图（来自 §2.1）：
<粘贴输出>

完整测试套件目录（带模块 / 特性 tag）：
<粘贴 tag 列表或文件路径>

平台关键路径清单：
<粘贴 — 例：发票接入 → 凭证生成 → 审计链路出账>

产出回归运行计划：
| 优先级 | 套件 | 估算运行时长 | 选用理由 | 覆盖哪条关键路径 |

规则：
- P0 = 覆盖被变更触及的关键路径的套件（必须跑）
- P1 = 覆盖 1-hop 调用方的套件
- P2 = 共享工具的烟雾测试套件
- 总运行时长要在团队回归预算内（默认 30 分钟；不清楚就问）
- 关键路径覆盖率达不到 100% 则升级，不允许部分跑

约束：输出里每一行都要引用变更影响地图的具体行作为选择依据。
```

### 2.3 C · 缺陷分流草稿

```
你是一位缺陷分流分析师。

Bug 工单：
<粘贴工单正文、报告人步骤、截图/日志引用>

近期相关 commit：
<粘贴受影响区域的 git log --oneline>

产出分流草稿：
- 严重度（P0 / P1 / P2 / P3）每档给 1 行定义理由
- 复现置信度（high / medium / low）— 基于步骤清晰度
- 怀疑组件 — 标文件或模块名
- 3 个候选根因假设（按可能性排序，每个配 1 个证伪测试）
- 建议责任人（研发团队 / 区域）
- 给报告人的 open question（≤ 3，问尖锐的）

约束：
- 复现步骤缺失或模糊则严重度默认 P3，直到被第二位工程师复现
- 假设必须可测 — "代码有 bug" 拒绝；"函数 F 第 N 行 OFF-BY-ONE" 可接受
```

### 2.4 A · 根因假设链

```
你是一位系统化调试分析师。

缺陷摘要（来自 §2.3）：
<粘贴>

§2.3 的最高假设：
<粘贴>

可得数据（日志、堆栈、近期变更）：
<粘贴>

产出根因链（≥ 3 跳）：
1. 症状：用户观察到什么
2. 近因：什么代码路径产生该症状（标 file:line）
3. 中因：什么状态 / 输入让该路径执行（标来源）
4. 根因：什么设计 / 数据假设被破坏（1 句话）
5. 证伪测试：哪一个测试能确认或打破这条链

约束：
- 每一跳都要引用具体证据（日志行、堆栈帧、commit SHA）
- 若可得证据不足以撑起 3 跳，停下并列出 "需要的证据" — 不要编造
- 末尾输出 1 行 "lessons learned"（测试 / 监控 / 文档里哪个缺口让这个漏出去）
```

## F3 · 合规与审计链路测试

### 3.1 P · 税规测试用例规划

```
你是一位业财税合规测试设计师。

被测税规：
<粘贴规则文本、政策通函或规则引擎配置>

规则生效日期与当前版本：
<粘贴 — YYYY-MM-DD 格式>

该规则现有测试用例（若有）：
<粘贴 file:test_name 列表，或 "暂无">

产出每条规则覆盖 ≥ 3 个分支的用例规划：
- 边界值测试（每个阈值 ≥ 1 个用例）
- 免税情形（≥ 1 个明确不适用规则的用例）
- 特殊条款（≥ 1 个针对规则特殊条款的用例）
- 历史兼容（≥ 1 个用例断言旧版本行为对老数据保留）
- 跨规则交互（≥ 1 个用例覆盖本规则与另一活跃规则的交互）

每条用例产出：
| 用例名 | 输入 fixture | 预期输出 | 审计链路断言 | 分支类型 |

约束：每条用例必须引用驱动预期输出的规则段落 / 条款。无锚点不允许意译。
```

### 3.2 D · 审计链路断言生成器

```
你是一位审计链路测试工程师。

源单据类型（发票 / 合同 / 收据 等）：
<粘贴 schema 或样例记录>

生成单据类型（凭证 / 账务记录 / 申报行 等）：
<粘贴 schema 或样例记录>

链路契约（哪些字段把源单据链接到生成单据）：
<粘贴 — 例：invoice_id → voucher.source_invoice_id>

产出审计链路断言套件：

每个生成单据：
1. ID 血缘断言：generated.source_id == source.id
2. 金额血缘：generated.amount == sum(source.line_amounts) — 或按规则规定
3. 税码血缘：generated.tax_code 由 source.tax_code + 适用规则版本派生
4. 时间戳血缘：generated.created_at >= source.issued_at（因果性）
5. 规则版本戳：generated.rule_version == active_rule_version_at(source.issued_at)

每条断言：
| 断言 | 测试名 | Fixture | 预期通过条件 | 什么会让它失败 |

约束：每条断言必须 1:1 从生成单据回溯到源单据。不允许 "计算后比较" 而不点名源字段。
```

### 3.3 C · 合规覆盖率审计

```
你是一位合规测试覆盖率审计员。

规则注册表（活跃规则 + 生效日期）：
<粘贴 — rule_id, name, effective_date>

测试用例清单（已有合规测试）：
<粘贴 — 测试文件路径、测试名映射到 rule_id>

产出缺口审计：
1. 零测试覆盖的规则（列 rule_id、name、effective_date、未覆盖天数）
2. 覆盖 < 3 个分支的规则（列出缺失的分支类型 — 边界 / 免税 / 特殊条款）
3. 测试过时的规则（断言基于早于当前生效版本的规则版本）
4. 未测的跨规则交互（作用域有重叠但没有联合测试用例的规则对）

每个缺口给严重度：
- P0：规则已生效且零测试 → 阻塞下次发布
- P1：规则 < 3 分支 → 下个 sprint 内必须关
- P2：版本过时 → 后续 2 个 sprint 内关

约束：不要编造规则名 — 只标输入注册表里存在的规则。
```

### 3.4 A · QA 周报撰写

```
你是一位 QA 周报编辑。

本周测试运行摘要（CI 结果）：
<粘贴 — 总跑次、通过率、最 flaky 的测试 top N>

缺陷工单流动（新增 / 已解 / 老化）：
<粘贴 — 按严重度分档>

覆盖率趋势：
<粘贴 — 行覆盖 / 分支覆盖 / 周环比 delta>

合规测试状态（有缺口的活跃规则）：
<粘贴 §3.3 的输出>

产出周报：
- 抬头："<YYYY-MM-DD> · QA 第 N 周 · 状态：GREEN / YELLOW / RED"
- 头条（1 句话 — 本周最大信号）
- 测试执行摘要（通过率、最 flaky 的 top 3 + 责任人）
- 缺陷流动（表格：按严重度的 新增 / 已解 / 待处理）
- 覆盖率趋势（文字图或表）
- 合规缺口（§3.3 的 P0 / P1 列表）
- 下周风险（≤ 3，每条含缓解责任人）
- 对团队的请求（≤ 2，具体且有时限）

约束：总字数 ≤ 600 字。超预算就删风险列表 — 绝不截断合规 P0 部分。
```

## 这些 prompt 怎么组合

| 工作流 | 调用顺序 |
|--------|---------|
| 新特性测试计划 | §1.1 → §1.2 → §1.3 → 关缺口 → §1.2 再跑 |
| 新 PR review 循环 | §2.1 → §2.2 → 跑选中套件 → §2.3 有缺陷则继续 → §2.4 跑头号缺陷 |
| 新税规上线 | §3.1 → §3.2 → §1.2（生成规划好的用例）→ §3.3 每周直到 100% 分支覆盖 |
| 缺陷调查 | §2.3 → §2.4 → 修 → §1.2 回归测试 → §2.2 确认无其他回归 |
| Sprint 收尾 | §1.3 + §3.3 → §3.4 周报 |

## 这些 prompt 不要用来干什么

- 实时生产问题调试 — 升级到 ops + 用 `systematic-debugging` skill，不是这些规划 prompt
- 性能 / 压力测试 — 不在 W4 范围；走专属性能套件 + APM 工具
- 安全渗透测试 — 不在 W4 范围；走安全团队 + 专属工具
- 探索性测试 — `claude-opus-chatstyle` 用于记笔记，无需脚本 prompt
