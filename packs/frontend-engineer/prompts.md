# Tool-Kit 02 · 提示词模板矩阵 · PDCA × 3 族

> 12 个产研常用提示词，按 PDCA（Plan / Do / Check / Act）× 3 族（后端 / 需求转文档 / 前端）二维分类。
> 与 `scenarios/` 下的详细 prompt 不冲突 — `scenarios/` 是完整长 prompt 链路，本文件是即用型短提示词速查。

## 矩阵全景

|  | Plan（规划） | Do（执行） | Check（核校） | Act（改进） |
|--|--------------|-----------|--------------|------------|
| **族 1 后端** | §1.1 接口 spec 草稿 | §1.2 联调脚本生成 | §1.3 失败用例分析 | §1.4 spec 与实现 diff |
| **族 2 需求转文档** | §2.1 PRD 反向问题 | §2.2 3 选项方案 | §2.3 多文档差异矩阵 | §2.4 SOT 合并发布 |
| **族 3 前端** | §3.1 设计 token 抽取 | §3.2 Tailwind 骨架 | §3.3 像素级自检 | §3.4 组件复用决策 |

> 每个 cell 是一段即用短 prompt（< 200 字），可直接粘到 Claude / ChatGPT / Codex 对话框。

## 族 1 · 后端 / 接口联调

### 1.1 P · 接口 spec 草稿

```
你是 OpenAPI 工程师。我要新增一个接口 <接口用途，1 句话>。
请输出 OpenAPI 3.0 spec 草稿（YAML），包含：
- path / method
- request schema（含字段类型 + 是否必填 + 示例）
- response schema（200 / 4xx / 5xx 各列一个）
- 字段命名遵循 <snake_case / camelCase> 约定
- 错误码用项目既有错误码体系

约束：不允许 "TBD" 占位；如真的不确定，列入 §1 反向问题清单给 PM。
```

### 1.2 D · 联调脚本生成

```
你是联调工程师。基于以下 OpenAPI spec：

<贴 OpenAPI YAML>

请生成 curl 联调脚本（bash），含：
- 正向用例 1 个（200）
- 边界用例 2 个（必填字段缺失 → 400）
- 异常用例 2 个（接口不存在 / 鉴权失败 → 401/404）
- 每个 case 含 assert（如检查返回字段 X 是否存在）

约束：URL 用 $API_BASE 环境变量；不要 hard-code prod 域名。
```

### 1.3 C · 失败用例分析

```
你是 debug 工程师。下面是联调失败的请求 + 响应：

REQUEST: <curl 命令>
RESPONSE: <实际返回 JSON 或 status code>
EXPECTED: <按 spec 应该返回什么>

请输出：
1. 错误归类（spec 不一致 / 接口 bug / 环境问题 / 数据问题 4 选 1）
2. 假设链（≤ 3 条，按优先级）
3. 每条假设的验证步骤
4. 建议下一步给前端 / 后端 / 测试 谁

约束：不要"试试 retry"或"加 try/catch"这种盲改。
```

### 1.4 A · spec 与实现 diff

```
你是接口审计员。下面是 spec 与实际接口返回的对比：

SPEC: <贴接口 schema>
ACTUAL: <实际 curl 返回>

请输出：
1. 字段级 diff 表（多 / 少 / 类型不符 / 命名不符）
2. 影响范围评估（破坏性 / 兼容性 / 文档误差）
3. 修复方案（改 spec / 改实现 / 改双方）
4. 谁该改 + 工作量预估（pd）

约束：不允许"两边都改改"的和稀泥结论。
```

## 族 2 · 需求转技术文档

### 2.1 P · PRD 反向问题清单

```
你是技术理解员。下面是 PM 给的 PRD：

<贴 PRD>

不要直接出方案。请输出：
1. 技术理解（3-5 句话，确认 PM 想表达什么）
2. 反向问题清单，分 3 级：
   - P0 阻塞（不答没法动手）3 个
   - P1 影响选型（如：是否要 1000+ 并发）3 个
   - P2 细节（如：日期格式）3 个
3. 我方初步假设（P2 不答时按此做）
4. 风险预警 1 句（若 P0 不答会怎样）

约束：不允许出"实现细节"。本步只是读懂 PRD。
```

### 2.2 D · 3 选项技术方案

```
你是方案架构师。基于已厘清的需求 <一句话总结>，请出 3 个方案。

CONSTRAINTS: 时间 <X 周> / 人力 <Y 人> / 主语言 <Z>

每个方案：
- 一句话定性
- 核心思路 2-3 句
- 工作量（前端 N pd / 后端 M pd / 测试 K pd）
- 优点 3 / 缺点 3 / 风险 3

方案 C 必须是"功能减半"对照组。

最后一张决策矩阵 + 推荐 1 个 + 不推荐 1 个 + 当 X 时回到 Z。
```

### 2.3 C · 多文档差异矩阵

```
你是文档归并员。下面有多份设计文档（已贴在下方），请输出差异矩阵：

DOCS: <doc A / B / C 内容>

1. 主题归集：每份文档实际覆盖什么主题
2. 重叠主题的差异矩阵：
   | 主题 | doc A 说法 | doc B 说法 | 冲突 | 裁决建议 + evidence |
3. 唯一覆盖主题清单（无冲突，可直接合入）
4. 风险标记（无来源 / 全冲突 章节列表）

裁决优先级：APPROVED > REVIEW > DRAFT；最新 > 最旧；与代码一致 > 不一致。
所有裁决必须给 evidence（哪份文档 + 哪段）。
```

### 2.4 A · SOT 合并发布

```
你是文档发布员。基于以下合并结构：

STRUCTURE: <贴上一步差异矩阵的合并结论>
TARGET: <Confluence / 飞书 / Wiki>

请输出：
1. 最终 SOT 文档（Markdown 完整版，含变更追溯 Appendix）
2. 发布脚本（git commit + 推送到目标平台 + 旧文档标 deprecated）
3. 通知模板（致目标群组，1 段，含旧地址 → 新地址引导）
4. 1 周观察期 follow-up（D+1 / D+3 / D+7 节点）

约束：旧文档标 deprecated 而非删除；保留可读 1 周。
```

## 族 3 · 前端开发

### 3.1 P · 设计 token 抽取

```
你是设计系统工程师。下面是 Figma 设计稿信息：

<贴 Figma Dev Mode 输出 / 截图描述 / 颜色字号列表>

请抽取 token，分类输出：
1. Color tokens（用途 / hex / 项目既有对齐 / 备注）
2. Typography tokens（用途 / size / weight / line-height）
3. Spacing tokens（8pt 网格制；偏移 ≥ 2px 单列）
4. Shadow / Radius tokens
5. 缺失 / 冲突 / 未使用既有 token 三段报告

约束：颜色用 hex；字号用 px 或 rem，不接受"差不多"。
```

### 3.2 D · Tailwind 骨架

```
你是前端工程师。基于以下 Figma 设计：

SCREEN: <截图描述或链接>
TOKENS: <贴上一步的 token 表>
FRAMEWORK: <React / Vue / 纯 HTML>

请输出：
1. 组件树（≤ 3 层嵌套）
2. 每个叶子组件的 Tailwind 骨架代码
3. Responsive 断点建议（sm / md / lg）
4. 缺失视觉细节清单（hover / disabled / loading 态 Figma 没给的）

约束：一律 Tailwind 工具类；不用任意值；缺失态不擅自补。
```

### 3.3 C · 像素级自检

```
你是前端 QA。请对当前页面做像素自检：

FIGMA: <Figma 链接>
DEPLOYED: <本地或预览 URL>
BREAKPOINTS: <要比对的断点>

请输出：
1. 检查项 checklist（颜色 / 字号 / 间距 / 圆角阴影 / 交互态 / 响应式 6 类）
2. 偏差清单（位置 / Figma 值 / 实际值 / 严重度 H/M/L / 修复建议）
3. 自检结论（H = 0 且 M ≤ 2 才算 PASS）
4. H/M 级偏差必须有截图对比

约束：严重度不允许全部标 L；自检通过 ≠ 设计走查通过。
```

### 3.4 A · 组件复用决策

```
你是前端架构师。基于既有组件库：

INVENTORY: <贴 src/components/index.ts 或 storybook 列表>
NEW SCREEN: <新页面需要哪些组件>

请输出：
1. 复用决策矩阵（新组件 / 既有候选 / 命中度 高/中/低/无 / 决策 / 工作量）
2. 改造组件的反向兼容评估（既有调用方影响 + 兼容性策略）
3. 新增组件命名 + 接口建议
4. 复用率统计（目标 ≥ 65%）
5. 风险标记（改造影响 ≥ 3 页 / 命名冲突 / 新增组件 < 2 调用方建议 inline）

约束：不允许全部"新增"；复用率 < 65% 必须解释。
```

## 用法约定

1. **PDCA 顺序不要跳**：跳过 Plan 直接 Do = 在假设上盖代码；跳过 Check 直接 Act = 不知道改对了没
2. **跨族组合**：复杂任务用多族多 P 串起来（如 PRD → 2.1 → 2.2 → 1.1 → 3.1）
3. **短 prompt 不够用时**：跳到 `scenarios/` 下找完整长 prompt 链路
4. **AI 输出永远是草稿**：必须人工 review；workshop 不是教 AI 替你做决定，是教 AI 把准备工作做完让你只做决定
