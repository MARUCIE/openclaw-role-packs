prototype
skills/prototype/SKILL.md

Rapid prototyping workflow. Skips normal standards to quickly validate a game concept or mechanic. Produces throwaway code and a structured prototype report.

是什么

帮你在拍板做一个新概念之前，用最低成本造一个能跑的原型来回答"这个想法到底成不成立"。它不追求工程质量，追求的是"一次性回答一个具体问题"，并出一份结构化的原型小结。

怎么用

  1. 用一句话写清楚这次原型要回答的核心问题，不要写成需求清单。
  2. 让它扫一眼当前的技术栈与已有素材，确认原型用什么底子搭最快。
  3. 给它定一个明确的时间盒（比如半天、一天），别让原型滑成小项目。
  4. 跑完原型后，让它输出一份结构化小结：核心问题、答案、证据、下一步建议。
  5. 如果结论是"成立"，再决定要不要立项；如果结论是"不成立"，把这份小结归档进决策日志。

架构图

flowchart LR
  A[一个核心问题] --> B[最小可行原型方案]
  B --> C[时间盒内搭建]
  C --> D[原型验证结果]
  D --> E[结构化决策小结]
stitch-design-pipeline
skills/stitch-design-pipeline/SKILL.md

Fully automated design workflow: PRD → Stitch prototype → Icon/Logo → Code translation. All steps automated end-to-end. User touches nothing.

是什么

帮你把一份 PRD（产品需求文档）一路自动跑到可交付的视觉原型与前端代码：先在 Stitch（Google 的 AI 设计工具）里出界面方案，再补图标和 Logo，最后把视觉稿翻译成代码。整条流水线无人值守，你只看入和出，中间不需要点鼠标。

怎么用

  1. 把一份相对完整的 PRD 交给它，确认里面有目标用户、关键场景、品牌调性这三件事。
  2. 让它跑 Stitch 那一段，先看它出的初版界面方向对不对，不对就让它整体重生成，不要逐屏改。
  3. 通过后让它顺着流水线把图标与 Logo 生成完，所有视觉素材保持同一套调性。
  4. 拿到视觉稿后让它翻译成前端代码，重点检查响应式与极端文案是否撑得住。
  5. 验收时按"PRD → 原型 → 视觉 → 代码"四段对照，看看是否仍然回到同一个产品意图。

架构图

flowchart LR
  A[PRD] --> B[Stitch 界面方案]
  B --> C[图标与 Logo]
  C --> D[视觉稿翻译成代码]
  D --> E[可交付的前端原型]
frontend-design
skills/frontend-design/SKILL.md

Use when building web UI from scratch: components, pages, landing pages, dashboards, HTML/CSS layouts, posters. Produces production-grade, aesthetically intentional code. NOT for backend logic. NOT for refactoring existing UI without redesign intent (use design-taste-frontend).

是什么

帮你从一张白纸起步，做出一个有判断力、有审美决策、不像 AI 默认输出的前端界面。它适合做新的组件、新的落地页、新的看板，把"我想要一个像样的页面"变成"可以直接发布的代码"。

怎么用

  1. 把目标用户、使用场景、品牌调性先讲清楚一句，让它知道这次的脾气是哪种。
  2. 让它先给一稿视觉方向的判断，包括字体、配色、节奏，确认大方向再动手写代码。
  3. 拿到第一版代码后，先在大屏和手机上各跑一遍，重点看响应式是不是真的能撑住。
  4. 把中文、英文、长文案、空状态都喂进去看一眼，确认极端内容不会让排版崩。
  5. 上线前让 design-taste-frontend 再过一遍品味体检，作为第二道把关。

架构图

flowchart LR
  A[场景与调性] --> B[视觉方向判断]
  B --> C[组件与页面落地]
  C --> D[多端与极端内容验证]
  D --> E[可发布的前端代码]
design-system
skills/design-system/SKILL.md

Guided, section-by-section GDD authoring for a single game system. Gathers context from existing docs, walks through each required section collaboratively, cross-references dependencies, and writes incrementally to file.

是什么

帮你给某一个具体的系统（比如"会员等级"、"积分规则"、"消息中心"）写出一份完整、自洽、研发能直接动手的系统设计文档。它不是让你填表，而是一段一段地陪你把每个必备章节聊清楚，并自动把已有上下文（依赖系统、术语表）串进来。

怎么用

  1. 先给它一个系统的短名字，让它去扫一遍现有文档，把相关上下文捞回来。
  2. 跟着它一节一节往下推进：先讲清楚"玩家/用户的体感"，再谈规则、公式、边界、依赖。
  3. 每过一节就让它落盘一次，避免聊到一半丢失进度。
  4. 遇到拿不准的依赖系统，把对应文档丢给它，让它自动交叉引用，不要凭记忆写。
  5. 最后让它跑一次完整自检，确认八大章节都齐了、术语统一了，再把文档给研发评审。

架构图

flowchart LR
  A[系统名] --> B[拉取相关上下文]
  B --> C[逐节协同撰写]
  C --> D[交叉引用依赖]
  D --> E[落盘系统设计文档]
design-taste-frontend
skills/design-taste-frontend/SKILL.md

Use for opinionated UI engineering standards: variance/motion/density

是什么

帮你给一个已经能跑起来的前端界面"加品味"——把那种一眼能看出是 AI 默认风格的页面，调成有节奏、有质感、有自己脾气的界面。它解决的是"功能没问题但就是不耐看"。

怎么用

  1. 把一个已经联调通的页面交给它，让它先做一遍口味体检，看变化度、动效、信息密度三项基线。
  2. 从体检报告里挑 3 到 5 处影响最大的改造点，别贪多，一次性全改没人能审。
  3. 让它给出具体的视觉规则修订建议，再让前端按这份建议一项一项落地。
  4. 每改一轮，回到真机上看观感，重点看长滚动页和复杂交互区，别只看截图。
  5. 把这次定下来的"品味基线"沉淀到团队的前端规范里，下一个页面起步就站在更高的水平。

架构图

flowchart LR
  A[已上线的界面] --> B[变化度/动效/密度体检]
  B --> C[挑 3-5 处改造点]
  C --> D[落地视觉规则]
  D --> E[更耐看的界面]
visual-style
skills/visual-style/SKILL.md

Create, extract, and apply portable visual design systems via visual-style.md

是什么

帮你把一套视觉风格沉淀成一份可复用的设计系统文件（visual-style.md），让同一套调性能跨 HeyGen 视频、HTML 幻灯片、Figma、纸质物料一起用。它解决的是"做完一次设计就一次性消耗掉了，没有沉淀"。

怎么用

  1. 先决定这次是从零创建，还是从一个网址、一段视频、一份 PDF 品牌手册里抽取风格。
  2. 让它把颜色、字体、版式、动效、情绪写进同一份 visual-style.md，不要分散在不同文档里。
  3. 把这份文件喂给后续的 HeyGen 视频、HTML 幻灯片或 Figma 工程，让它们共用一套调性。
  4. 每完成一个产物，把它放回 visual-style 的画廊里做一次自检，看视觉是否被稀释。
  5. 把这份风格文件纳入项目的设计系统目录，让新加入的同事一上手就站在统一基线。

架构图

flowchart LR
  A[网址/视频/品牌手册] --> B[抽取或新建风格]
  B --> C[visual-style.md]
  C --> D[视频/幻灯片/Figma 共用]
  D --> E[统一调性的多端产物]
impeccable-design
skills/impeccable-design/SKILL.md

[PLANNED] Placeholder - not yet implemented

是什么

帮你把一个已经做完功能的界面，打磨到"挑不出毛病"的程度——把对齐、留白、文案、状态、边界等容易在细节上掉链子的地方，一项一项过一遍。它的目标是让发布前的最后一公里不留遗憾。

怎么用

  1. 把准备上线的页面交给它，让它先把所有可见状态截一遍（空、加载、出错、超长内容、极端权限）。
  2. 让它按"对齐 - 留白 - 字号层级 - 文案口径 - 极端状态"这五项依次扫一遍。
  3. 拿到的问题清单按"必修 / 选修"分两栏，先把必修项发给前端排进当前迭代。
  4. 修完一轮再让它复扫一次，确认没有新引入的回归。
  5. 把这次的修订要点沉淀成团队的"上线前自检清单"，下次直接照着跑。

design-review
skills/design-review/SKILL.md

Reviews a game design document for completeness, internal consistency, implementability, and adherence to project design standards. Run this before handing a design document to programmers.

是什么

帮你在把设计文档（PRD/PRP/GDD 等产品需求与设计稿）交给研发之前，先做一次八维度体检：完整性、内一致性、可实现性、是否符合团队规范。把"模糊的设计稿"变成"工程师拿到就能动手的交付物"。

怎么用

  1. 在每次评审会前，把待审的设计文档路径喂给这个技能，让它先跑一遍自动体检。
  2. 拿到八个维度的红黄绿评分后，先看红灯项，那是工程师最容易卡住的地方。
  3. 把"内一致性问题"清单贴到评审会议纪要里，逐条和设计、研发对齐口径。
  4. 修订完文档后，再跑一次，确保所有红灯转绿，再走交付流程。
  5. 把发现的高频缺漏沉淀到团队的设计文档模板里，让下一次起草时就不再犯。

架构图

flowchart LR
  A[设计文档草稿] --> B[八维度体检]
  B --> C[内一致性检查]
  C --> D[红黄绿评分]
  D --> E[交付前评审报告]

多 CLI 安装支持（Claude Code / Codex / Gemini / Hermes / OpenClaw）

install.sh v5.1 起支持 多 CLI agent 自动适配。同一份 install.sh 可以装到五个目标，靠 --agent 标志或自动检测决定落点：

CLI Agent 默认目录 安装命令
Claude Code~/.claude/curl -fsSL .../packs/product-manager/install.sh | bash
Codex CLI~/.codex/curl -fsSL .../packs/product-manager/install.sh | bash -s -- --agent=codex
Gemini CLI~/.gemini/curl -fsSL .../packs/product-manager/install.sh | bash -s -- --agent=gemini
Hermes agent~/.hermes/curl -fsSL .../packs/product-manager/install.sh | bash -s -- --agent=hermes
OpenClaw~/.openclaw/curl -fsSL .../packs/product-manager/install.sh | bash -s -- --agent=openclaw
其它 / 自定义$INSTALL_DESTINSTALL_DEST=/your/dir curl -fsSL ... | bash

优先级: INSTALL_DEST 环境变量 → --agent=<X> 标志 → OPENCLAW_AGENT 环境变量 → 自动检测（按 ~/.claude → ~/.codex → ~/.gemini → ~/.hermes → ~/.openclaw 顺序）→ 默认 claude。

artifact 的 dst 路径（skills/ · agents/ · CLAUDE.md · AGENTS.md）是 Claude + Codex + Gemini + Hermes + OpenClaw 间的共享约定。CLAUDE.md 在 Codex/Gemini/Hermes/OpenClaw 上是惰性文件（不读取，也无害）。
